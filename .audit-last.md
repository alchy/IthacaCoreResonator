### 2. Thread Safety

#### **[BUG] DspChain parameter setters race with RT thread**

`CoreEngine::setLimiterThreshold/Release/Enabled` and `setBBEDefinition/BassBoost` (GUI thread) call into `DspChain` setters which write non-atomic members of `Limiter` and `BBE` (e.g. threshold float) while `DspChain::process()` runs from the RT callback. This is a data race / UB per the C++ standard. On x86 aligned float writes are hardware-atomic so it won't corrupt in practice, but the code provides no formal guarantee.

**Fix:** Make the DSP parameter floats `std::atomic<float>`, or use a double-buffer / message-queue approach.

#### **[UB] `voices_[m].active` in `PianoCore::getVizState()` vs RT thread**

`getVizState()` reads `voices_[m].active` (plain `bool`) from the GUI thread while `processBlock` writes it from the RT thread (`piano_core.cpp:393, 409`). Same UB class as above. The `ISynthCore` comment justifies "direct float write is acceptable on x86 as a pragmatic choice" — this needs to be extended explicitly to `bool` members read in `getVizState`, or `active` should be `std::atomic<bool>`.

### 3. Real-Time Safety in processBlock

#### **[PERF BUG] Denormals in decay state**

`piano_core.cpp:335-336`:
```cpp
ps.env_fast *= ps.decay_fast;
ps.env_slow *= ps.decay_slow;
```
and `noise_env *= noise_decay` (line 355) decay toward zero unconditionally. The skip threshold at line 338 only skips oscillator *output*, it does not prevent these state variables from reaching the denormal range (~1e-38). On SSE2 hardware without DAZ/FTZ mode, each denormal multiply is ~100× slower than a normal float multiply. With 128 voices × 60 partials × 2 envelopes each, this is a significant CPU load after long sustained notes.

No FTZ/DAZ is set anywhere in the audio callback.

**Fix:** Either set `_mm_setcsr(_mm_getcsr() | 0x8040)` at the start of `audioCallback`, or add a floor check:
```cpp
if (ps.env_fast < 1e-9f) ps.env_fast = 0.f;
```

#### **[Minor] `v.ndist = std::normal_distribution<float>(0.f, 1.f)` in RT thread**

`piano_core.cpp:234` — called from `initVoice` → `handleNoteOn` → MIDI queue drain inside `processBlock`. `std::normal_distribution` assignment is implementation-defined; on all major STL implementations it does not allocate (it stores two floats and a flag), but this is not guaranteed by the standard. Low practical risk, but worth noting.

### 4. Bugs and Design Flaws

#### **[MINOR BUG] `resonator_engine.cpp` is orphaned dead code with a second `MINIAUDIO_IMPLEMENTATION`**

The file defines `#define MINIAUDIO_IMPLEMENTATION` and is not included in any CMake target (was the old engine, now superseded by `CoreEngine`). If it's ever accidentally added back to a target that also links `core_engine.cpp`, you get multiple-definition linker errors on all miniaudio symbols. The file should be deleted or moved to an `archive/` directory.

#### **[Minor] `EQ_POINTS` constant visible in `biquad_eq.h` but defined elsewhere**

`biquad_eq.h:31` uses `EQ_POINTS` (parameter to `design()`), but the constant is defined in `note_params.h`. This is an implicit dependency — the header compiles only if `note_params.h` was included first. It already is via `#include "note_params.h"` at line 2, so this is fine. Not a bug, but noted.

#### **[Doc bug] CMakeLists comment is wrong**

`CMakeLists.txt:31-32`:
```cmake
# Core engine (no miniaudio, no audio device; shared across non-RT uses)
    synth/core_engine.cpp       # contains MINIAUDIO_IMPLEMENTATION
```
The comment says "no miniaudio, no audio device" but the very next line says `core_engine.cpp` contains `MINIAUDIO_IMPLEMENTATION`. The original intent may have been that `CORE_ENGINE_SOURCES` was designed to be split, but it never was. Misleading.
